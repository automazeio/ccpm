# Task: Routing Decision Rules

## Description
Document comprehensive task classification criteria for routing decisions between Claude API and local LLM. Create a decision tree that clearly defines when to use each model based on task characteristics, complexity, and context.

## Acceptance Criteria
- [ ] Documents code generation patterns (keywords, file types, complexity indicators)
- [ ] Documents planning patterns (PRD, epic, architecture, design docs)
- [ ] Provides concrete examples of each task type with explanations
- [ ] Clear decision tree for classification with visual representation
- [ ] Used as reference by routing hook implementation
- [ ] Includes edge cases and fallback rules
- [ ] Performance characteristics documented (when to prefer speed vs quality)

## Technical Details

### Implementation Approach
- Create `ccpm/rules/local-llm-decision-tree.md` as the canonical reference
- Define task type taxonomy (code generation, planning, review, mixed)
- Document keyword patterns and file extension triggers
- Include complexity scoring rubric
- Provide examples from real CCPM workflows

### Key Considerations
- Balance between cost savings and quality
- Handle ambiguous tasks that could go either way
- Consider task context (part of larger workflow)
- File size and complexity thresholds
- User override mechanisms

### Files Affected
- `ccpm/rules/local-llm-decision-tree.md` (new file)

### Decision Factors to Document
1. **Task Type Indicators**
   - Keywords: "generate", "implement", "create file", "write code"
   - File operations: new files, code refactoring, boilerplate
   - Planning keywords: "PRD", "epic", "architecture", "design"

2. **Complexity Scoring**
   - Low: Boilerplate, simple CRUD, config files
   - Medium: Business logic, integrations, refactoring
   - High: Architecture decisions, complex algorithms, cross-cutting concerns

3. **Quality Requirements**
   - Critical: Security, payment logic, core algorithms
   - Standard: Feature implementation, utilities
   - Low: Tests, documentation, examples

4. **Context Factors**
   - Part of epic workflow
   - Standalone task
   - Interactive session
   - Batch processing
